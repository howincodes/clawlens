# ClawLens v2 Roadmap — Future Features

Features to build after v1.1 is stable and adopted.

---

## v2.0 — Intelligence Layer

### Knowledge Graph / Expertise Mapping
- Track which topics each dev successfully resolves with AI
- Build per-dev expertise profile: "Bob knows Kafka, Alice knows auth"
- When a dev starts prompting about a topic → suggest "Sarah solved this last week"
- Requires: enough historical data (2-4 weeks), semantic analysis of prompts

### Git/PR Integration
- Connect GitHub/GitLab via OAuth
- Map Claude sessions → commits → PRs
- Dashboard shows: "This feature was 40% AI-generated"
- Track AI-generated code quality: bugs, reverts, review comments
- Requires: GitHub API integration, branch/commit detection from session context

### Prompt Quality Coach
- Real-time scoring: clarity, specificity, context provided
- Inline suggestions: "Add error logs for better results"
- Before-after comparison: show how prompt improvements lead to faster resolution
- Requires: fast model call per prompt (latency concern — may need to be async)

### Response Caching
- Detect semantically similar prompts across team
- Cache successful responses
- Suggest cached response before hitting Claude API
- Potential 30-70% cost savings for repetitive queries
- Requires: embedding model for semantic similarity, cache invalidation strategy

### Prompt Template Library
- Best prompts auto-detected from high-completion-rate sessions
- Shared across team as templates
- Tagged by: language, framework, task type
- Admins can curate, devs can browse
- Requires: template extraction logic, template management UI

## v2.1 — Policy & Governance

### Prompt Policy Engine
- Block prompts containing secrets (beyond detection — actually block the prompt)
- Block copying proprietary code to external models
- Regex + AI-based detection
- Custom rules: "Never send files from /internal/ directory"
- Requires: PreToolUse or UserPromptSubmit returning block decision based on content analysis

### Model Routing Rules
- Auto-route: simple prompts → Haiku, complex → Sonnet, architecture → Opus
- Admin defines rules or uses AI-based auto-routing
- Requires: prompt complexity classifier, managed-settings model override per-prompt

### Approval Workflows
- Large prompts (>5K tokens) → require manager approval before sending
- High-cost model usage → require justification
- Requires: async approval mechanism, possibly Slack integration

### Project-Level Policies
- Different limits per project/repo
- "Production repo → no Opus, only Sonnet with 10/day limit"
- Requires: project detection from cwd, per-project limit rules

## v2.2 — Predictive & Autonomous

### Usage Forecasting
- "At current pace, Alice will hit her limit by 3pm"
- "Team will exceed monthly budget by March 28th"
- Requires: time series analysis on usage data

### Burnout Detection
- Late-night heavy usage + repeated failures = potential burnout signal
- Sensitive: must be opt-in, visible only to the dev themselves + manager
- Requires: careful UX, privacy considerations

### Auto-Pause on Abuse
- Automatically pause users who hit limits repeatedly
- Configurable: after N blocks in M hours → auto-pause for X hours
- Requires: auto-pause rules in team settings

### Anomaly Detection (advanced)
- ML-based: learn each dev's usage pattern, flag outliers
- "Bob's usage spiked 300% — possible credential sharing?"
- Requires: baseline modeling per user

## v2.3 — Integrations

### Slack App (not just webhook)
- Interactive: `/clawlens status` in Slack
- Approve/deny requests from Slack
- Daily standup replacement: AI brief posted to channel
- Requires: Slack app registration, OAuth, interactive messages

### JIRA/Linear Integration
- Map sessions to tickets
- "Time spent with AI on PROJ-123: 2.5 hours"
- Auto-update ticket with AI activity summary
- Requires: OAuth + API integration

### SSO/OIDC
- Google Workspace, Okta, Auth0
- Replace password-based admin auth
- Per-user dashboard access (devs see their own data)
- Requires: OIDC provider integration, role-based access

### Email Reports
- Daily/weekly briefs via email (not just Slack/Discord)
- Requires: email service (SendGrid, SES, or SMTP)

## v2.4 — Scale & Enterprise

### PostgreSQL Migration Path
- Keep SQLite for <50 devs
- Offer Postgres adapter for larger teams
- Same queries, swappable backend
- Requires: database abstraction layer

### Multi-Admin Roles
- Admin: full access
- Viewer: read-only dashboard
- Manager: see their team only (subset of users)
- Requires: RBAC, team hierarchy

### Air-Gapped Deployment
- No external network calls (summary AI runs locally via Ollama/llama.cpp)
- All binaries served from internal network
- SOC2-friendly audit logs
- Requires: local LLM support, offline binary distribution

### Event Schema Versioning
- Client v1.0 and v2.0 can coexist
- Server handles multiple event formats
- Requires: schema registry, version negotiation

### Horizontal Scaling
- Separate ingestion from processing
- Queue (NATS/Kafka) between client events and database writes
- CDN for dashboard static files
- Requires: architecture redesign for >100 teams

## Data We Should Start Collecting Now (for v2)

Even if we don't build these features yet, collect the data so it's available when we do:

| Data | How | For which v2 feature |
|------|-----|---------------------|
| Full tool_input (not just 200 chars) | Store in tool_event (configurable) | Policy engine, code quality analysis |
| Git branch name | From session cwd + `git branch --show-current` | Git/PR integration |
| File paths modified | From PostToolUse on Write/Edit tools | Code tracking, project mapping |
| Subagent usage | SubagentStart/Stop hooks (add in v1.2) | Team workflow analysis |
| Task completions | TaskCompleted hook (add in v1.2) | Productivity measurement |
| Prompt embeddings | Compute on server from prompt text | Knowledge graph, duplicate detection, caching |

## Competitive Positioning for v2

| Competitor | Their Focus | Our Differentiator |
|-----------|------------|-------------------|
| Helicone/Langfuse | API-level LLM observability | We see the developer's local context, not just API calls |
| GitHub Copilot Dashboard | Copilot-specific analytics | We work with Claude Code, deeper prompt analysis |
| Anthropic Console | Account-level usage | We do per-developer, per-project, with rate limiting |
| Datadog/New Relic | General APM | We're purpose-built for AI developer tools |

**Our moat:** We're the only tool that captures the full developer-AI interaction lifecycle (prompt → tools → response → outcome) at the local level, with AI-powered analysis. API observability tools only see the API call — we see what the developer asked and what happened next.
