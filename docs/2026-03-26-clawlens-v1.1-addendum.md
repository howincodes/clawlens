# ClawLens v1.1 Addendum — New Features + Architecture Fixes

**Date:** 2026-03-26
**Applies to:** The base spec at `2026-03-26-clawlens-design.md`

---

## Architecture Changes (from review)

### Go Binary (confirmed)

Client and server are both Go. No Node.js dependency.

- **Client:** single Go binary (~8MB), 6 platform builds
- **Server:** single Go binary (~15MB) with embedded React dashboard via `//go:embed`
- **SQLite:** pure Go via `modernc.org/sqlite` (no CGO)
- **Cross-compile:** `GOOS=darwin GOARCH=arm64 go build`

Hooks call the binary directly: `"command": "/usr/local/bin/clawlens hook prompt"`

### Data Flow (hybrid — local queue + batch sync)

Only 2 synchronous server calls:
1. `session-start` → check-in, get config, auto-update check
2. `prompt` → rate limit gate (the only blocking call per prompt)

Everything else writes to **local SQLite queue** and batch syncs:

```
prompt → local SQLite (<1ms)
pre-tool → local SQLite (<1ms)
stop → local SQLite (<1ms)
session-end → local SQLite (<1ms)

Background goroutine (every 30s, configurable from server):
  → read unsynced events from local DB
  → POST /api/v1/sync-batch → server
  → mark events as synced locally
```

### Simplified API (single event endpoint)

Replace 8 hook endpoints with one:

```
POST /api/v1/event
{
  "type": "prompt" | "tool" | "stop" | "stop_error" | "session_end",
  "session_id": "...",
  "data": { ... }
}
```

Keep separate endpoints only for:
- `POST /api/v1/session-start` — check-in + config + auto-update
- `POST /api/v1/prompt` — synchronous rate limit gate
- `POST /api/v1/sync-batch` — batch event upload
- `POST /api/v1/register` — install code exchange
- `GET /api/v1/health`

### Merge usage_event into prompt table

Drop `usage_event` table. Add `credit_cost INTEGER` to `prompt` table. Rate limiting queries against `prompt` table with index on `(user_id, timestamp, credit_cost)`.

### Reduce to 6 hooks for v1

1. **SessionStart** — check-in, device sync, auto-update check
2. **UserPromptSubmit** — rate limit gate + prompt recording
3. **PreToolUse** — kill/pause enforcement + tool tracking (local only)
4. **Stop** — turn completion, response recording
5. **StopFailure** — Anthropic rate limit / error tracking
6. **SessionEnd** — session close

Drop for v1: PostToolUse, SubagentStart, SubagentStop, TaskCompleted (add in v2).

### Local Client Event Queue Table

```sql
-- Client-side only (in local SQLite)
CREATE TABLE event_queue (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  event_type TEXT NOT NULL,
  payload TEXT NOT NULL,
  created_at DATETIME NOT NULL,
  synced BOOLEAN DEFAULT FALSE
);
CREATE INDEX idx_event_queue_synced ON event_queue(synced, created_at);
```

### Batch Sync Interval

Default: 30 seconds (configurable from server via session-start response).

Server can tell clients to back off under load. Client caches the interval in local config.

## v1.1 Features

### 1. Stuck Detection

Detect when a developer is struggling — 5+ prompts on the same topic within 20 minutes with similar tool calls failing.

**How it works:**
- Server analyzes recent prompts per user per session
- Groups by semantic similarity (same project_dir + overlapping keywords)
- If prompt_count > 5 in 20 min with had_error > 50% → stuck
- Triggers: webhook alert to team lead, dashboard notification

**New table:**

```sql
CREATE TABLE alert (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  team_id TEXT NOT NULL,
  user_id TEXT REFERENCES user(id) ON DELETE CASCADE,
  type TEXT NOT NULL,        -- stuck | anomaly | secret_detected | rate_limit_anthropic
  severity TEXT NOT NULL,    -- info | warning | critical
  title TEXT NOT NULL,
  details TEXT,              -- JSON with context
  resolved BOOLEAN DEFAULT FALSE,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_alert_team ON alert(team_id, created_at);
```

**Server-side detection job:**
- Runs every 60 seconds
- Checks each active session for stuck patterns
- Creates alert + sends webhook if configured

### 2. Client-Side Secret Scrubbing

Before prompts leave the dev's machine, scan for secrets and optionally redact.

**Patterns detected (regex):**
- API keys: `(sk-|pk-|api[_-]?key|token)[a-zA-Z0-9_-]{20,}`
- AWS: `AKIA[0-9A-Z]{16}`
- Connection strings: `(postgres|mysql|mongodb|redis)://[^\s]+`
- Private keys: `-----BEGIN (RSA |EC |)PRIVATE KEY-----`
- Generic secrets: `(password|secret|passwd)[\s]*[=:]\s*['"][^'"]{8,}`

**Behavior (configurable via team settings):**
- `"secret_scrub": "redact"` — replace detected secrets with `[REDACTED-API-KEY]` before sending
- `"secret_scrub": "alert"` — send prompt as-is but create a `secret_detected` alert
- `"secret_scrub": "off"` — no scanning

**Implementation:** In the Go client, scan prompt_text before writing to local queue or sending to server. Fast regex, adds <1ms per prompt.

Add to team settings:
```json
{
  "secret_scrub": "redact",  // "redact" | "alert" | "off"
}
```

### 3. Smart Model Suggestions

Analyze prompt patterns and suggest when devs use expensive models unnecessarily.

**How it works:**
- Server tracks: model used + prompt complexity (length, tool count, session type)
- If a user sends >N short/simple prompts on Opus → flag as "could use Sonnet/Haiku"
- AI summary includes model efficiency score + specific suggestions

**Dashboard:**
- Per-user "Model Efficiency" score (0-100)
- Suggestion: "Alice used Opus for 15 simple queries this week — Haiku would've saved $X"
- Team-level: "Your team could save 30% by routing simple queries to Haiku"

No new tables — computed from existing `prompt` data in analytics service.

### 4. Duplicate Work Detection

Detect when two devs ask Claude similar things about the same codebase.

**How it works:**
- Server groups prompts by project_dir + time window (same week)
- AI analyzes pairs of prompts from different users on same project
- If semantic similarity > threshold → create alert

**Runs in:** AI summary job (alongside per-user summaries)

**Alert:** "Alice and Bob both asked Claude about the auth middleware this week — they might benefit from syncing up."

### 5. ROI Estimation

Estimate time/money saved by AI usage.

**How it works:**
- Each completed session (prompt → tool calls → stop) = one "task"
- Estimate time saved based on task type:
  - Debugging session (many tool calls, error patterns) → estimated 30 min saved
  - Feature development (Write/Edit tools) → estimated 1-2 hours saved
  - Simple query (no tools, short response) → estimated 5 min saved
- AI summary includes ROI section: "Claude saved Alice ~4 hours this week at $12 cost = $3/hour"

**Dashboard:**
- Team ROI card: "Estimated X hours saved this month at $Y cost"
- Per-user ROI breakdown

Computed in analytics service from prompt + tool_event data.

### 6. Daily/Weekly Intelligence Brief

AI-generated report that replaces standup meetings.

**Daily brief (per user):**
```

What she worked on:
- Debugged auth middleware (2 sessions, 14 prompts)
```

**Delivery:**
- Dashboard page (AI Summaries)

## Security Improvements (from review)

### Signed Binary Updates

- Client verifies SHA-256 checksum of new binary before replacing
- Server stores checksums per version
- Binary replacement is atomic: download to same directory → `rename()` (atomic on same filesystem)
- Version pinning: admin sets `target_version` in team settings — clients only update to that version

### Auto-Update Versioning

```json
// session-start response includes:
{
  "update": {
    "available": true,
    "version": "1.2.0",
    "sha256": "abc123...",
    "url": "/bin/clawlens-darwin-arm64",
    "required": false
  }
}
```

Admin controls from dashboard:
- Target version
- Force update toggle (blocks devs until updated)
- Client version inventory (see who's on what version)

### Auth Improvements

- JWT expiry: 24 hours (require re-login)
- Login rate limiting: max 5 attempts per minute per IP
- Auth token rotation: `POST /api/admin/users/:id/rotate-token`
- Pluggable auth middleware (for future SSO/OIDC)

### Fail-Open on Server Down

Changed from fail-closed to **fail-open with degraded mode** for the rate limit gate:

```
prompt hook:
  1. Try server /api/v1/prompt (3s timeout)
  2. Server responds → use server decision
  3. Server timeout → evaluate limits LOCALLY from cached config
  4. No cache at all → ALLOW (don't block devs from working)
  5. Log the offline event for reconciliation on next sync
```

Kill/pause remains fail-closed (checked locally from cache).

### Session Orphan Cleanup

Server job runs every 10 minutes:
- Find sessions with no activity for >30 min
- Set `ended_at = NOW()`, `end_reason = "timeout"`

### Prompt Truncation

Store max 10,000 characters per prompt. If longer:
- Truncate at 10,000 chars
- Set `prompt_truncated = TRUE` on the prompt record
- AI summaries process the full prompt before truncation (on the client, before sending)
